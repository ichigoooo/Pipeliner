# Pipeliner：技术栈选择说明

## 文档目的

本文档用于单独记录当前阶段对 Pipeliner 技术栈的选择结论、取舍理由和分阶段演进策略，避免技术选型讨论继续分散在主设计文档与对话记录中。

本文档当前记录两类内容：

- 已确认设计：已经达成一致、可作为 V1 实施基础的部分。
- 后续演进：当前不阻塞 V1，但会影响后续 UI/UX 升级路线的部分。

---

## 一、当前结论（已确认）

当前确认，Pipeliner 的 V1 技术路线为：

- 后端核心使用 `Python` 实现。
- 优先完成 Runtime、协议模型、Artifact 管理、回调接收与状态推进等后端核心能力。
- 当前阶段的界面只使用 `Python` 提供最简单可用的 GUI 或管理界面，不把前端体验作为 V1 的核心目标。
- 在验证后端核心跑通、协议合理、运行闭环成立之后，再使用 Web 技术重做或增强 UI/UX。

一句话概括：

> V1 先用 Python 做对后端最有利的实现，界面先够用；等核心验证通过后，再用 Web 技术升级体验层。

---

## 二、为什么当前选择 Python（已确认）

当前确认，Python 更适合 Pipeliner 当前阶段的核心问题形态：

- 协议对象较多，适合用强结构化数据模型快速落地。
- 文件系统操作较多，适合处理 run root、artifact payload 和本地工作目录。
- 需要频繁和脚本、子进程、Agent 工具链打交道。
- Validator Skill 中天然会包含大量脚本化校验逻辑。
- 当前更重要的是先验证 Runtime 与协议设计，而不是先做复杂前端交互。

因此，Python 更适合承担：

- Runtime
- callback API
- workflow spec / callback payload / manifest 数据模型
- artifact 管理
- 本地执行编排
- CLI 与调试工具

---

## 三、为什么当前不优先投入 Web 前端（已确认）

当前确认，并不是 Web 前端不重要，而是它暂时不应成为 V1 的前置阻塞项。

原因是：

- 当前最需要验证的是系统核心是否成立。
- 核心问题在于协议、Runtime、Artifact 流转与验收闭环。
- 如果在这些能力尚未跑通前就优先投入复杂前端，会显著分散开发精力。
- 一旦底层协议后续继续调整，前端很容易跟着频繁返工。

因此，当前阶段更合理的策略是：

- 先让后端核心收敛并跑通。
- 界面先做到“可查看、可触发、可调试”。
- 等后端设计证明稳定后，再投入更高质量的 UI/UX。

---

## 四、V1 当前建议的技术栈（已确认）

当前建议的 V1 技术栈如下：

- 语言：`Python 3.12+`
- API / Runtime：`FastAPI`
- 协议建模与校验：`Pydantic v2`
- 持久化：`SQLite`
- ORM：`SQLAlchemy 2.x`
- 迁移：`Alembic`
- CLI：`Typer`
- 包管理与运行：`uv`
- 测试：`pytest`
- Artifact 存储：`local_fs`

这套选择的设计意图是：

- 用最适合结构化协议与本地运行时的技术，把核心后端尽快做稳。
- 用最小依赖完成 V1，不提前引入不必要的分布式复杂度。

---

## 五、V1 界面策略（已确认）

当前确认，V1 界面策略为：

- 界面目标是“最简单可用”，而不是“最终产品级 UI/UX”。
- 当前只需要满足：
  - 查看 workflow
  - 查看 run 状态
  - 查看节点状态
  - 查看 artifact 基本信息
  - 触发基础操作
- 不把复杂工作流编辑器、拖拽画布、精细视觉体验纳入 V1 必做范围。

这里进一步确认：

- 当前阶段允许界面比较朴素。
- 只要它不妨碍验证核心协议和 Runtime，就足够。

---

## 六、后续演进路线（已确认）

当前建议的演进路线如下：

### 阶段 1：后端核心先跑通

重点完成：

- `workflow spec` 模型与校验
- `node callback payload` 模型与接收
- `artifact manifest` 模型与解析
- Runtime 最小状态推进
- 本地 run root 与 artifact 存储

### 阶段 2：提供最简单可用 GUI

重点完成：

- workflow 列表和详情
- run 列表和详情
- 节点状态查看
- artifact 基本查看
- 简单操作入口

### 阶段 3：验证核心后再升级 UI/UX

在核心验证通过后，再考虑使用 Web 技术重做或增强前端，例如：

- 更友好的节点卡片视图
- 更好的流程图视图
- 工作流编辑器
- 更完整的 authoring 体验
- 更成熟的用户交互与视觉设计

---

## 七、当前明确不在 V1 范围内的内容（已确认）

为了保持当前阶段聚焦，以下内容不作为 V1 的前置目标：

- 高完成度的 Web 前端
- 拖拽式工作流编辑器
- 产品级视觉设计
- 复杂实时协作界面
- 高级交互动效与深度 UX 打磨

这些内容后续都可以做，但当前不应阻塞核心实现。

---

## 八、当前结论

截至目前，技术栈选择可以被概括为：

> 用 Python 优先完成 Pipeliner 的后端核心与最小可用界面。
>
> 当前阶段把精力集中在协议、Runtime 与 Artifact 闭环上。
>
> 等核心得到验证后，再使用 Web 技术升级 UI/UX。

更简洁地说：

```text
V1 = Python-first core, minimal GUI now, richer web UI later
```
