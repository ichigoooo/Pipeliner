## Context

当前所有 Claude Code 的调用只返回最终结果或错误，缺少对执行过程的可视化。开发者无法看到实时终端输出，导致问题定位成本高且反馈链路不完整。此变更要求在所有 Claude Code 调用入口提供可展开的只读终端视图，以便观察执行过程与输出。 

## Goals / Non-Goals

**Goals:**
- 在所有 Claude Code 调用入口提供统一的可展开终端视图。 
- 终端输出实时追加，完成后可回看。 
- 仅展示，不提供任何交互控制。 
- 展示内容与 CLI 调用时一致，包含过程输出与可用的思维链/流程信息。 
**Non-Goals:**
- 不提供终端输入、暂停/继续、重试等交互能力。 
- 不重构现有 Claude Code 调用协议或命令模板。 
- 不在本次变更中引入新权限体系或审计策略。 
## Decisions

1. **统一的 Claude 调用日志模型**
   - 选择在 Claude Code 调用的统一封装层写入“终端输出流”记录，所有入口共用。 
   - 通过统一输出流避免各 UI 自行拼接或遗漏，保证“所有调用入口”一致性。 
2. **前端组件化终端视图**
   - 提供可复用的 `ClaudeTerminalPanel` 组件，默认折叠，支持展开查看。 
   - 组件展示流式输出与最终状态，独立于具体页面布局。 
3. **流式输出优先，轮询兜底**
   - 首选 SSE/WS 等推送机制展示实时输出。 
   - 若环境受限，保留轮询接口作为兜底。 
4. **输出渲染策略**
   - 以“终端文本”为主视图，必要时支持 ANSI 转换以保持 CLI 一致性。 
   - 对超长输出进行分段加载，避免前端卡顿。 
## Risks / Trade-offs

- **日志过长影响性能** → 通过分段写入、前端虚拟化与输出长度上限控制。 
- **思维链/过程输出可能包含敏感信息** → 仅对具备权限的开发者入口展示，并预留脱敏/关闭开关。 
- **部分入口遗漏接入** → 在 Claude 调用封装层统一注入输出流，避免分散实现。 
## Migration Plan

1. 在 Claude Code 调用封装层新增输出流采集与持久化。 
2. 提供流式读取接口，并在前端增加终端面板组件。 
3. 逐个入口接入终端面板，验证输出一致性与性能。 
4. 若需回滚，仅隐藏面板并停用流式接口，保留日志采集与调用协议不变。 
## Open Questions

- “思维链/流程信息”具体指 Claude Code 提供的哪类输出？是否需要可配置开关？ 
- 日志保留时长与最大体积限制是多少？ 
- 是否需要渲染 ANSI 颜色，还是保持纯文本即可？
