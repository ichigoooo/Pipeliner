## Why

当前运行体验存在明显断层：用户在 Studio 中点击“启动运行”后，常常只能看到一个 run 被创建，却无法判断系统是否已经真正开始执行、正在处理哪个节点，或者只是停在等待态。这会让正常运行看起来像卡死，削弱对工作流系统的信任，也增加不必要的人工排障和重复点击。

## What Changes

- 将 workflow run 的默认启动行为从“仅创建 run”调整为“创建后立即自动驱动执行”。
- 为运行详情页补充面向操作员的实时活动流，展示节点排队、执行器启动、校验器启动、回调到达和终态变化。
- 在运行页补充 workflow 流程图视图，高亮当前活跃节点，并在点击节点后以分 tab 方式展示终端输出、产物、回调和原始数据。
- 在运行中禁用冲突性的手动 `Drive` 操作，同时保留手动驱动作为高级控制入口。
- 扩展运行态聚合 API，返回 driver 状态、当前焦点节点和活动流数据。

## Capabilities

### New Capabilities

None.

### Modified Capabilities

- `run-operations`: 启动 run 的默认行为、运行中可观测性，以及自动驱动与手动驱动的操作语义将被增强。

## Impact

- 后端运行控制：`POST /api/runs`、run drive 协调逻辑、run overview 聚合逻辑。
- 前端 Studio：workflow 版本页的启动行为、run 详情页的流程图化布局与实时反馈。
- 类型与测试：`RunOverview` / 启动请求模型扩展，新增自动驱动与活动流相关测试。
